# 需求变更与扩展登记

> **用途**：登记所有「不在当前开发阶段实现，但最终必须交付」的需求。
> **原则**：需求不进代码，但必须有可追溯的书面登记，防止口头约定最后被遗忘。
> **维护规则**：每条需求一个登记项，含编号、来源、状态、描述、预留扩展点、计划实现步骤、验收标准。

---

## 登记项 #1：多环境配置区分（dev / test / prod）

| 字段 | 内容 |
|------|------|
| **登记编号** | REQ-001 |
| **登记日期** | 2026-06-24 |
| **来源** | 用户在 Step 1 讨论中提出 |
| **当前状态** | ⏳ 待实现（计划在 Step 15 落地） |
| **优先级** | 高（项目最终交付的硬性要求） |

### 需求描述

支持 **dev（开发）/ test（测试）/ prod（生产）三种运行环境**，实现配置与敏感信息的彻底隔离：

1. **三套独立配置**：每种环境有独立的业务配置（默认模型、存储后端类型、超时重试参数等）。
2. **三套独立敏感信息**：每种环境有独立的 API Key、数据库密码等（存入各自的 `.env`，互不污染）。
3. **运行时一键切换**：通过环境变量（如 `APP_ENV=dev|test|prod`）或启动参数切换环境，无需改动代码。
4. **配置隔离验证**：切换环境后，加载的配置与数据源必须真正隔离（如 dev 的数据库不会被 prod 误写，dev 的密钥不会泄漏到 prod）。

### 为什么不在当前阶段（Step 1~14）实现

- 遵循 **YAGNI 原则**（You Aren't Gonna Need It，不过度设计）：当前阶段只有一个运行环境，引入多环境会增加复杂度、拖慢逐step验证节奏。
- 这是「**运行/部署层面**」的增强，不依赖任何业务功能，放最后实现最合理，且能反向检验「配置与代码解耦」做得是否到位。

### 预留的扩展点（Step 1~14 落地）

1. `config.yaml` 采用清晰的分层结构，字段命名按「将来可被环境覆盖」的方式组织。
2. 当前阶段为单一环境运行，但 `app.env` 字段以注释形式预留。
3. 配置加载代码（Step 2 的 `config_manager`）后续支持「基础配置 + 环境覆盖配置」的合并机制即可，无需重构数据结构。

### 计划实现步骤

**Step 15：多环境配置区分实现与验证**（新增于《实施步骤计划》v1.2，详见该文档）

- 引入环境区分机制：`APP_ENV` 环境变量 + `config.{env}.yaml` 覆盖配置 + `.env.{env}` 敏感值
- 改造配置加载逻辑：基础配置 + 环境覆盖合并
- 三环境分别配置并冒烟验证

### 验收标准（必须全部满足）

- [ ] 设置 `APP_ENV=dev` 时，程序加载 dev 配置（SQLite + 开发用便宜模型），冒烟测试通过：创建用户 → 发起对话 → 导出会话
- [ ] 设置 `APP_ENV=test` 时，程序加载 test 配置（独立测试数据库），`uv run pytest` 全部通过
- [ ] 设置 `APP_ENV=prod` 时，程序加载 prod 配置（MySQL + 正式模型或模拟），核心功能冒烟通过
- [ ] 三个环境的 `.env` 互相隔离，切换环境时不会读到其它环境的密钥
- [ ] 三种环境的数据库互相隔离，dev 运行不会写入 prod 数据库
- [ ] 文档（README / architecture）中说明如何切换环境

## 登记项 #2：多模型切换

| 字段 | 内容 |
|------|------|
| **编号** | REQ-002 |
| **当前状态** | ⏳ 延后完善（已有初版能力，待统一配置与测试） |
| **优先级** | 高 |

### 需求描述

允许用户在多个 OpenAI-compatible 模型之间切换，并保证模型名称、接口地址、密钥来源、
温度和最大输出长度等参数可配置。项目已有模型选择原型，本需求登记的是后续的配置统一、
能力探测、错误提示和完整验收，不在 Step 1 重构现有实现。

### 为什么当前阶段不实现

Step 1 的目标是建立可运行、可追踪的工程骨架。完整模型切换依赖配置加载、模型客户端封装、
UI 交互和真实或模拟接口测试，属于后续业务开发范围；当前调整可能破坏已有 chatbot 功能。

### 预留扩展点

- `config.yaml` 已提供 `models.default`、`models.available`、温度和最大 token 配置。
- `.env.example` 已预留通用及供应商 API 配置变量。
- 现有 `config/settings.py`、`services/llm_service.py` 和模型选择 UI 可作为后续整合入口。

### 计划实现步骤

1. 建立统一模型配置对象并校验默认模型是否可用。
2. 将供应商密钥和接口地址从环境变量安全注入模型客户端。
3. 完善会话级模型切换、能力兼容检查和错误回退。
4. 增加模拟客户端测试与至少一个真实供应商的人工冒烟测试。

### 验收标准

- [ ] 配置中至少两个模型时，界面可选择并正确调用目标模型。
- [ ] 切换模型不会丢失当前会话历史，且新消息记录实际使用的模型。
- [ ] 缺少密钥、模型不存在或接口不兼容时给出明确提示。
- [ ] 模型配置与真实密钥均不硬编码在代码或可提交文件中。

## 登记项 #3：会话历史持久化

| 字段 | 内容 |
|------|------|
| **编号** | REQ-003 |
| **当前状态** | ⏳ 延后完善（已有 SQLite 原型，待存储抽象与迁移测试） |
| **优先级** | 高 |

### 需求描述

可靠保存用户、会话和消息历史，支持重启后恢复、按用户隔离、查询和后续存储后端切换。
现有 SQLite 代码予以保留；本需求关注后续的存储接口、数据迁移、并发安全和完整验收。

### 为什么当前阶段不实现

Step 1 只定义存储配置结构，不新增数据模型或迁移机制。存储抽象和数据库兼容涉及业务接口、
事务策略、测试数据及迁移方案，超出工程初始化范围。

### 预留扩展点

- `config.yaml` 已预留 `sqlite`、`mysql` 和 `file` 三类存储配置。
- 现有 `database/` 模块可逐步封装为统一仓储接口。
- `data/` 已被 Git 忽略，避免本地会话和数据库进入仓库。

### 计划实现步骤

1. 定义会话、消息和用户的存储接口及领域数据结构。
2. 将现有 SQLite 操作迁移到接口实现并补充事务处理。
3. 增加 MySQL 与文件存储实现或适配器。
4. 编写重启恢复、用户隔离、并发写入和迁移测试。

### 验收标准

- [ ] 应用重启后可恢复原有会话及消息顺序。
- [ ] 不同用户不能读取或修改彼此的会话。
- [ ] 切换受支持的存储类型时，上层聊天服务无须改动业务逻辑。
- [ ] 数据库文件、测试数据和连接密码不会提交到 Git。

## 登记项 #4：用户预设 Prompt 管理

| 字段 | 内容 |
|------|------|
| **编号** | REQ-004 |
| **当前状态** | ⏳ 待实现（Step 1 仅提供系统内置预设） |
| **优先级** | 中 |

### 需求描述

在系统内置 Prompt 之外，允许用户创建、编辑、复制、启用和删除自己的 Prompt 预设，
并在会话中选择。用户预设需与账号或用户标识关联，不能覆盖只读的系统内置预设。

### 为什么当前阶段不实现

Step 1 只需要交付 `config/presets.yaml` 中的内置数据。用户级管理还需要数据模型、权限边界、
输入校验和 UI，提前实现会扩大当前阶段范围。

### 预留扩展点

- `config/presets.yaml` 使用稳定的 `id`、名称、描述和 `system_prompt` 字段。
- 现有 `prompts/` 模块可增加系统预设与用户预设的合并读取接口。
- 会话记录可继续保存预设标识，后续支持版本追踪。

### 计划实现步骤

1. 定义 Prompt 预设模型、字段校验和系统/用户归属规则。
2. 增加用户预设的持久化与增删改查服务。
3. 在聊天界面加入预设管理和选择入口。
4. 增加越权访问、重名、删除引用中预设等边界测试。

### 验收标准

- [ ] 用户可创建、编辑、复制和删除自己的预设。
- [ ] 系统内置预设可选择但不能被普通用户修改或删除。
- [ ] 用户只能看到自己的私有预设和公共系统预设。
- [ ] 已开始的会话能明确追踪所使用的预设或其快照。

## 登记项 #5：聊天记录导出

| 字段 | 内容 |
|------|------|
| **编号** | REQ-005 |
| **当前状态** | ⏳ 待实现 |
| **优先级** | 中 |

### 需求描述

支持将单个会话按时间顺序导出为 Markdown、JSON 等格式，包含会话标题、模型、预设、
消息角色和时间信息，并生成安全、可识别的文件名。

### 为什么当前阶段不实现

导出依赖稳定的会话数据模型、格式规范、隐私规则和界面交互。Step 1 只预留输出目录与文件名模板，
不实现文件生成业务。

- `config.yaml` 已提供 `export.dir` 和 `export.filename_template`。
- 导出目录位于被忽略的 `data/` 下，默认不会进入 Git。
- 现有聊天服务可在后续暴露只读的完整会话查询接口。

### 计划实现步骤

1. 定义与存储后端无关的会话导出数据结构。
2. 实现 Markdown 和 JSON 渲染器及安全文件名生成。
3. 加入界面下载入口和敏感信息过滤策略。
4. 编写中文、特殊字符、空会话和长会话导出测试。

### 验收标准

- [ ] 可将指定会话导出为 Markdown 和 JSON。
- [ ] 导出消息顺序、角色、内容和元数据与原会话一致。
- [ ] 文件名符合模板且不会产生路径穿越或非法字符问题。
- [ ] 导出内容不包含 API Key、数据库密码等运行环境秘密。

## 登记项 #6：异常处理与重试机制

| 字段 | 内容 |
|------|------|
| **编号** | REQ-006 |
| **当前状态** | ⏳ 延后完善（已有基础提示，待分类重试与可观测性） |
| **优先级** | 高 |

### 需求描述

对模型超时、限流、网络故障、无效密钥、数据库错误和配置错误进行分类处理；只对可恢复错误
执行有上限的重试，并向用户返回友好、可操作且不泄露秘密的信息。

### 为什么当前阶段不实现

可靠重试需要明确错误分类、幂等边界、退避策略、日志脱敏和自动化测试。Step 1 仅提供
`llm.timeout`、`llm.max_retries` 与日志配置，不改变现有调用链。

### 预留扩展点

- `config.yaml` 已预留 LLM 超时和最大重试次数。
- `config/logging.yaml` 已提供控制台、滚动文件和第三方库降噪配置。
- 现有 `services/llm_service.py` 与 `services/chat_service.py` 可增加统一异常映射层。

### 计划实现步骤

1. 定义配置、认证、限流、超时、网络和存储异常类型。
2. 为可恢复错误实现指数退避、抖动和最大次数限制。
3. 确保消息写入与重试满足幂等要求，避免重复回复或重复记录。
4. 增加日志脱敏、用户提示及故障注入测试。

### 验收标准

- [ ] 超时和临时网络错误按配置重试，超过上限后停止。
- [ ] 认证失败、参数错误等不可恢复异常不会盲目重试。
- [ ] 重试不会产生重复用户消息或重复助手回复。
- [ ] 日志和界面错误信息不暴露 API Key、密码或完整连接串。
